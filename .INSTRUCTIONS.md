# .INSTRUCTIONS.md — How to handle a task in this project

When the user gives a new task in the Crash-On Analysis project, follow this
exact workflow every time. This keeps the work consistent and the memory
current across sessions.

## Workflow (do these in order)

1. **Read the task** the user provided.
2. **Read this file** (`.INSTRUCTIONS.md`) for the standing procedure.
3. **Read the memory file** (`MEMORY.md`) to recover context:
   - project goal, architecture, Socket.IO protocol
   - Cloudflare `cf_clearance` / Playwright bypass decision
   - current dashboard cards, endpoints, known bugs, how to run
4. **Inspect current code** before changing anything:
   - `backend.py` (Flask + Playwright collector + SQLite + API endpoints)
   - `templates/index.html` (dashboard, Chart.js, live polling)
   - `crashon.db` (SQLite; table `rounds`) — inspect with:
     `python -c "import sqlite3; db=sqlite3.connect('crashon.db'); print(db.execute('SELECT * FROM rounds ORDER BY rowid DESC LIMIT 3').fetchall())"`
5. **Make the change**:
   - Mimic existing patterns (event handling, metric computation, endpoints,
     dashboard cards, Chart.js usage).
   - Keep the principle: **server `houseProfit` / `availableLiquidity` are NOT
     trusted** — compute House Profit locally as `total_bet_value − amount_won`.
   - Preserve full-precision liquidity display (`fmtLiq`).
   - Always add `global connected` in `on_event` if you touch connection state.
   - Import `request` from flask when adding query params.
6. **Verify** the change end-to-end:
   - Stop any running python, then `run.bat` (or `python backend.py`).
   - Hit the relevant endpoint(s), e.g.:
     `http://localhost:8080/api/db/outcomes`
     `http://localhost:8080/api/db/predictions?window=30`
     `http://localhost:8080/api/db/rounds?limit=50`
     `http://localhost:8080/api/status`
   - Confirm `backend.err` shows no errors and data is flowing.
   - For HTML/dashboard changes, confirm the page renders and updates live.
7. **Update `MEMORY.md`** with what changed:
   - new endpoints, new dashboard cards, new behavior, new bugs fixed,
     anything that future sessions need to know.
   - Keep the file accurate — it is the source of truth for resuming.

## Quick reference
- Python: `C:\Users\PM_User\AppData\Local\Programs\Python\Python312\python.exe`
- Playwright Python pinned to **1.56.0** (matches chromium-1194 build).
- Run: `run.bat` → open `http://localhost:8080`
- Stop backend: kill the `python` process.
- DB table: `rounds` (round_id PK, crash_point, server_seed, public_seed,
  total_bets, total_bet_value, amount_won, house_edge, house_profit,
  available_liquidity, cashouts JSON, crashed_at, recorded_at).

## Rules
- Do not trust the server's reported liquidity / house profit.
- Do not break the Playwright-in-page WebSocket approach (needed for
  Cloudflare). A plain websocket-client gets HTTP 400.
- Keep the dashboard no-cache so HTML edits show immediately.
- Update memory after every task — even small ones.
